Ngày 03: User admin và profile

Hoàn thiện user/profile như hầu hết hệ thống đều cần: xem mình, sửa profile, admin search, deactivate.

Mục tiêu

  • Learner cập nhật được profile của chính mình.
  • Admin tạo/mời được learner mới theo flow Keycloak + access-control + profile.
  • Admin list/search user có paging.
  • Admin deactivate/reactivate user theo rule rõ.

Hai luồng tạo tài khoản

Luồng Khi nào dùng API/entrypoint Kết quả bắt buộc
Admin provision Backoffice tạo learner cho lớp/course. POST /api/admin/v1/users Keycloak account, access-control identity link, role STUDENT, profile trong user-management.
Self-registration Visitor tự tạo learner account ở learner web/mobile. http://localhost:3000/register -> Keycloak registration Keycloak account, gateway auto-provision access-control identity, role STUDENT.
Rule SSO khi test: nếu browser thường đang login admin thì test self-registration bằng incognito/browser profile khác. Cùng browser profile trong cùng realm courseflow chỉ có một SSO user active.

Cách impl

  1. Đọc model user/profile/status trong user-management-service; phân biệt account identity từ Keycloak với profile/directory data trong app.
  2. Chốt chính sách tạo tài khoản: mini stack có hai luồng chuẩn, admin provision ở admin web và self-registration ở learner web qua Keycloak.
  3. Thiết kế request tối thiểu cho admin tạo learner: email, fullName, temporaryPassword, requirePasswordChange, sendSetupEmail; không cho client tự gửi userId hoặc role tuỳ ý.
  4. Implement hoặc hoàn thiện flow admin provision: tạo user ở Keycloak, gọi access-control để link issuer+subject và gán role STUDENT, tạo profile mặc định trong user-management-service, trả DTO không chứa password/token.
  5. Implement hoặc kiểm tra flow learner self-registration: Keycloak tạo account, callback về learner web, gateway resolve/provision identity lần đầu qua access-control và gán STUDENT.
  6. Chốt status lifecycle tối thiểu: ACTIVE, DISABLED; nếu có PENDING/LOCKED thì chỉ dùng khi đã có rule rõ.
  7. Thiết kế API admin list/search users: filter q/email/status/role, paging, sort ổn định; không trả toàn bộ dataset không giới hạn.
  8. Implement repository query bằng index-friendly condition; với search user directory nhỏ có thể dùng DB query, nhưng tránh trả field nhạy cảm.
  9. Thêm hoặc kiểm tra index DB cho email/status/displayName nếu list/search cần dùng thường xuyên; migration có tên rõ và rollback note.
  10. Implement deactivate/activate user: admin-only, không cho admin tự disable chính mình nếu đó là admin cuối cùng trong hệ thống demo.
  11. Khi disable user, cập nhật status trong user-management-service; Keycloak disable có thể là phase sau, nhưng tài liệu evidence phải nói rõ boundary hiện tại.
  12. Implement learner update profile: chỉ user tự sửa profile của mình; admin update profile người khác nếu có endpoint riêng thì phải check permission.
  13. Validate unique email nếu service quản lý email; nếu email lấy từ Keycloak thì không cho sửa email qua profile API.
  14. Chuẩn hóa DTO: admin list có role/status; learner profile chỉ có thông tin public cần hiển thị.
  15. Xử lý lỗi: user không tồn tại 404, status transition sai 400, student gọi admin endpoint 403, duplicate email 409.
  16. Evidence bắt buộc: admin tạo learner mới, learner xuất hiện trong Keycloak và admin user list, learner login được bằng password tạm thời, self-registration pass trong incognito/profile khác, admin search user, disable student2, student2 gọi API bị chặn hoặc status DISABLED, student update profile và read-back đúng.

Kết quả cần đạt

  • GET /api/v1/users/me chạy ổn sau login.
  • POST /api/admin/v1/users tạo hoặc mời learner mới theo boundary Keycloak/access-control/user-management.
  • PUT /api/v1/users/me/profile cập nhật display name/avatar/bio.
  • GET /api/admin/v1/users?q=... trả paging đúng.
  • User inactive không được coi là active learner trong các feature sau.

Evidence

POST /api/admin/v1/users
Authorization: Bearer <admin_token>
Body: {
  "email": "learner01@courseflow.local",
  "fullName": "Learner 01",
  "temporaryPassword": "password",
  "requirePasswordChange": false,
  "sendSetupEmail": false
}
Expected: 201, user ACTIVE, role STUDENT

Login learner web bằng learner01@courseflow.local/password
Expected: vào dashboard, GET /api/v1/users/me trả đúng user

Self-registration:
- Mở incognito hoặc browser profile khác nếu browser thường đang login admin.
- Vào http://localhost:3000/register.
- Đăng ký user mới qua Keycloak.
Expected: callback về learner web, user có role STUDENT trong access-control.

PUT /api/v1/users/me/profile
Authorization: Bearer <student_token>
Body: {"displayName":"Demo Learner","bio":"Learning Java"}
Expected: 200, data cập nhật

GET /api/admin/v1/users?q=student&page=0&size=10
Authorization: Bearer <admin_token>
Expected: có student, có paging metadata

Negative:
- displayName quá dài -> 400
- duplicate email khi admin tạo learner -> 409 hoặc lỗi rõ từ Keycloak/user-management
- student gọi admin list -> 403
