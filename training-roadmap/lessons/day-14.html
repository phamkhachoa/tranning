Ngày 14: Integration QA và manual evidence

Gom các feature lại thành flow end-to-end và chứng minh bằng request/response/log/DB.

Manual QA Integration Evidence

Sơ đồ impl

flowchart TB Feature["Feature evidence"] --> Happy["happy path request"] Feature --> Error["error/forbidden path"] Feature --> ReadBack["read-back query"] Happy --> Response["HTTP status + response body"] Error --> Response ReadBack --> DBLog["DB row / service log"] Response --> Conclusion["PASS/FAIL + root cause"] DBLog --> Conclusion Conclusion --> BugList["bug list + priority"]

Mục tiêu

  • Mỗi nhóm có evidence cho feature mình làm.
  • End-to-end flow chính chạy qua gateway.
  • Lỗi blocker được fix hoặc ghi root cause rõ.

Cách impl

  1. Chốt một kịch bản E2E duy nhất để QA: admin/instructor tạo course có media, publish, student search, enroll, học, submit assignment, làm quiz, nhận grade/notification/chat.
  2. Reset dữ liệu demo hoặc dùng course mới có slug unique để evidence không lẫn với dữ liệu cũ.
  3. Kiểm tra auth đầy đủ: admin, instructor, student, student2; mỗi role có token/session riêng.
  4. Khi cần mở nhiều user đồng thời, dùng browser profile khác nhau hoặc normal window + incognito; cùng realm Keycloak không login hai user khác nhau trong cùng browser profile.
  5. Chạy health check gateway và service chính: user-management, access-control, course, enrollment, assignment, quiz, gradebook, notification, chat, search, media.
  6. Kiểm tra platform: Keycloak realm, Eureka dashboard có registered services, MinIO bucket có object, Kafka Connect connector RUNNING, Elasticsearch index có document.
  7. Happy path qua gateway, không gọi trực tiếp service: tất cả URL API dùng http://localhost:28080.
  8. Error path authz: student gọi admin publish bị 403, student2 xem submission/grade của student1 bị 403.
  9. Error path business: enroll draft/archived course bị reject, duplicate enrollment không tạo bản thứ hai, submit assignment quá attempt/lock bị reject nếu rule có.
  10. Read-back path: sau mỗi write quan trọng phải có GET/list chứng minh dữ liệu đã lưu đúng.
  11. Kiểm tra eventual consistency: sau publish chờ Debezium/search index, sau grade chờ notification/outbox; evidence ghi rõ wait/retry hợp lý.
  12. Gom curl/browser evidence vào một file hoặc mục demo: request, response chính, expected, actual, pass/fail.
  13. Tạo backlog phase sau: test automation, observability dashboard, performance search, transcode worker thật, fine-grained RBAC.

Kết quả cần đạt

  • Auth/profile/course/enrollment/progress/assessment/notification/search chạy qua gateway.
  • Mỗi feature có ít nhất một negative case.
  • Evidence có token/user/endpoint/status/response chính.
  • Danh sách bug còn lại có priority rõ.

Evidence

Template:
Feature: <name>
User: <admin|instructor|student>
Request: <method/path/body>
Expected: <status + response chính>
DB/log: <row/log cần kiểm tra>
Result: PASS/FAIL + root cause nếu fail
