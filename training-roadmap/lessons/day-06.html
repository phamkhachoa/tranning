Ngày 06: Enrollment transaction

Learner đăng ký course với rule published-only, chống duplicate và có transaction rõ.

Enrollment Transaction Idempotency

Sơ đồ impl

flowchart LR LearnerWeb["Learner web"] --> EnrollAPI["POST /api/v1/enrollments"] MobileApp["Mobile app"] --> EnrollAPI EnrollAPI --> Auth["CurrentUser + STUDENT"] Auth --> CourseCheck["course-service
course must be PUBLISHED"] CourseCheck --> Tx["transaction"] Tx --> Unique["unique learnerId + courseId + ACTIVE"] Unique --> DB[("enrollment DB")] Tx --> Outbox[("optional outbox event")] DB --> Response["ACTIVE enrollment"] Unique -->|duplicate| Existing["return existing or idempotent error"]

Mục tiêu

  • Learner enroll được course PUBLISHED.
  • Không tạo trùng ACTIVE enrollment khi retry/double click.
  • Admin/instructor xem được roster theo course.

Cách impl

  1. Đọc enrollment-service entity/repository: enrollment id, courseId, studentId, status, idempotency key, createdAt/updatedAt.
  2. Đọc client gọi course-service để kiểm tra course metadata/pricing/status; enrollment-service không query database course.
  3. Thiết kế API enroll: body courseId, optional idempotencyKey; studentId lấy từ CurrentUser.
  4. Trước khi insert, gọi course-service/internal metadata để đảm bảo course PUBLISHED và purchasable nếu có pricing rule.
  5. Chống duplicate active enrollment bằng unique constraint hoặc transaction check + unique index; cùng student/course active chỉ có một record.
  6. Nếu request retry cùng idempotencyKey, trả lại enrollment cũ thay vì tạo record mới.
  7. Nếu course archived/draft, trả 409/400 rõ; nếu student chưa auth trả 401; nếu instructor enroll với role learner tùy rule training.
  8. Implement list my enrollments: filter current student, include course summary nếu cần bằng API/read model, không join cross DB.
  9. Implement admin list by courseId/status: staff-only, paging, sort mới nhất trước.
  10. Phát outbox event enrollment.created sau khi transaction commit path lưu outbox; notification sẽ dùng ở ngày 11.
  11. Đặt boundary progress: enrollment-service sở hữu enrollment state, course-service có thể sở hữu item progress nếu hiện code đang vậy; ghi rõ contract giữa hai bên.
  12. Evidence bắt buộc: student enroll published course, retry không duplicate, enroll draft bị reject, admin list thấy enrollment, DB có đúng một active record.

Kết quả cần đạt

  • Enroll course PUBLISHED thành công.
  • Enroll course DRAFT bị chặn.
  • Gọi enroll hai lần không sinh hai dòng ACTIVE.
  • Roster admin có paging và chỉ dành cho operator/instructor hợp lệ.

Evidence

POST /api/v1/enrollments
Authorization: Bearer <student_token>
Body: {"courseId":"<published_course_id>"}
Expected: 200/201, status ACTIVE

Repeat same request
Expected: không có duplicate ACTIVE row

Negative:
- course DRAFT -> 400/409
- chưa login -> 401
