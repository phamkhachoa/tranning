Ngày 18: API contract và release readiness

Đóng lại khóa bằng contract rõ, lỗi thống nhất, versioning và checklist deploy được.

OpenAPI Error contract Versioning Release checklist

Mục tiêu

  • Không để web/app đoán response field.
  • Chuẩn hóa error body và HTTP status cho các service.
  • Biết cách version API mà không phá client cũ.
  • Chuẩn bị release note ngắn: migration, config, rollback, known issues.

Cách impl

  1. Chọn 5 API quan trọng nhất đã làm: users/me, create course, enroll, submit quiz/assignment, search courses.
  2. Với mỗi API, ghi method/path/auth role/request/response/error cases vào api-contract.html hoặc OpenAPI nếu đã thêm dependency.
  3. DTO request không nhận field server-owned: userId, ownerId, graderId, status transition actor, createdAt.
  4. DTO response không trả password, internal token, storage secret, raw MinIO object key nếu không cần.
  5. Chuẩn lỗi: validation 400, unauthenticated 401, forbidden 403, not found 404, duplicate/conflict 409, rate limit 429, dependency failure 502/503.
  6. Đọc ApiExceptionHandler: mọi service nên trả cùng shape ErrorDto.
  7. Thêm errorCode ổn định cho lỗi quan trọng để web/app dịch message mà không parse text.
  8. Versioning: breaking change đi sang /api/v2 hoặc thêm field backward-compatible; không đổi nghĩa field cũ âm thầm.
  9. Release checklist: env var mới, migration mới, dữ liệu seed demo, compose service health, rollback note nếu migration fail.
  10. Evidence cuối khóa: một bảng API contract, một lỗi validation, một lỗi forbidden, một lỗi conflict, một release note ngắn.

Kết quả cần đạt

  • API surface đủ rõ cho admin web, learner web và mobile app.
  • Error contract giống nhau giữa các service chính.
  • Release note nói được cần deploy gì, kiểm tra gì, rollback thế nào.
  • Backlog phase sau có test automation/OpenAPI generation/CI thay vì chỉ “làm tiếp”.
