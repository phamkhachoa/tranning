Ngày 17: Cache, rate limit và resilience

Giảm tải hệ thống nhưng vẫn giữ correctness ở database/service boundary.

Redis Rate limit Timeout Retry

Sơ đồ impl

flowchart LR Client["Web/App"] --> GW["Gateway"] GW --> RL["Rate limiter
Redis token bucket"] RL --> API["Protected API"] API --> Cache["Cache-aside
short TTL"] Cache --> Redis[("Redis")] Cache --> DB[("Source DB")] API --> ClientCall["Service-to-service client"] ClientCall --> Timeout["timeout + bounded retry"] Timeout --> Downstream["Downstream service"] Timeout --> Fallback["fail closed or graceful error"]

Mục tiêu

  • Biết chỗ nào được cache, chỗ nào không được cache.
  • Thiết kế rate limit ở gateway cho API dễ bị spam.
  • Thêm timeout/retry có giới hạn cho service-to-service call.
  • Không dùng Redis làm nguồn đúng sai cho nghiệp vụ quan trọng.

Cách impl

  1. Chọn một endpoint read-heavy để cache: public course catalog, course detail, search suggest hoặc RBAC permission decision ngắn hạn.
  2. Xác định cache key không chứa dữ liệu nhạy cảm: version prefix + tenant/scope + filter normalized + page/size.
  3. Đặt TTL ngắn: catalog 30-120 giây, permission decision 5-30 giây; không cache response chứa token hoặc profile private.
  4. Invalidation: khi publish/archive/update course thì delete cache key liên quan hoặc chấp nhận TTL ngắn và ghi rõ trade-off.
  5. Với rate limit, chọn key theo userId nếu đã auth, fallback IP cho public endpoint; student không share cùng bucket với admin.
  6. Rate limit ở gateway trước khi vào service để giảm tải; API trả 429 với message rõ.
  7. Với service-to-service client, đặt timeout nhỏ hơn timeout toàn request; ví dụ access-control 1500ms, token converter 800ms.
  8. Retry chỉ cho lỗi transient và idempotent GET/resolve; không retry command tạo dữ liệu nếu thiếu idempotency key.
  9. Fail closed cho auth/access decision: không gọi được access-control thì deny hoặc 503, không tự allow.
  10. Gắn metric: cache hit/miss, rate limit rejected, downstream timeout/retry.
  11. Evidence phải chứng minh: cache hit nhanh hơn hoặc ít query hơn, quá rate limit trả 429, downstream timeout không treo request.

Kết quả cần đạt

  • Redis được dùng cho cache/rate-limit, không dùng làm source of truth.
  • Gateway có plan rate limit cho public/search/auth-heavy API.
  • Service client có timeout rõ và lỗi trả về có kiểm soát.
  • Người học giải thích được vì sao retry sai có thể nhân đôi dữ liệu.

Evidence

Redis:
docker exec courseflow-training-redis redis-cli keys '*courseflow*'

Cache evidence:
GET /api/v1/courses?q=java
GET lại cùng query
Expected: cache hit metric/log hoặc thời gian giảm

Rate limit evidence:
for i in $(seq 1 30); do curl -i http://localhost:28080/api/v1/search/courses?q=java; done
Expected: một số request trả 429 nếu đã bật limiter

Timeout evidence:
stop access-control-service hoặc chỉnh URL sai
Expected: protected API fail closed, không treo quá timeout đã đặt
