Ngày 01: Keycloak login và current user

Hoàn thiện đường đi tối thiểu: web/app login Keycloak -> gateway verify token -> user-management trả current user.

Mục tiêu

  • Login được admin web, learner web và mobile app client bằng tài khoản Keycloak demo.
  • Giải thích được khác nhau giữa JWT ký bằng shared secret và OIDC token verify bằng JWKS/public key.
  • Gọi được GET /api/v1/users/me qua gateway.
  • Gateway set đúng X-User-Id, X-User-Roles, X-Internal-Authorization.

Trọng tâm auth của ngày 01

Cách impl

  1. Chạy stack bằng docker compose -f docker-compose.training.yml up -d --build; kiểm tra keycloak, api-gateway, user-management-service đều healthy trước khi viết code.
  2. Mở Keycloak admin console http://localhost:18080, xác nhận realm courseflow, client courseflow-api, users demo admin/instructor/student/student2 đã được import.
  3. Mở http://localhost:18080/realms/courseflow/.well-known/openid-configuration và chỉ ra issuer, jwks_uri. Mở tiếp /protocol/openid-connect/certs để thấy danh sách public keys.
  4. Đọc ConfiguredGatewayExternalTokenVerifier: xác định gateway dùng NimbusReactiveJwtDecoder.withJwkSetUri, không verify Keycloak token bằng shared secret.
  5. Đọc OAuth2AudienceValidator: giải thích vì sao token ký đúng nhưng sai aud vẫn phải bị reject.
  6. Đăng nhập web/app hoặc lấy token bằng flow dev; mọi request chính đi qua gateway với Authorization: Bearer <access_token>, không gọi service trực tiếp từ web/app.
  7. Đọc JwtAuthenticationGatewayFilter: xác định các bước verify issuer, audience, JWKS, strip header giả mạo, map identity demo và mint internal JWT local.
  8. Đọc route user-management-current-user trong gateway để hiểu /api/v1/users/me rewrite thành endpoint nội bộ của user-management-service.
  9. Trong user-management-service, đọc controller current user/profile, DTO profile và entity user/profile tương ứng; chỉ để controller parse request và gọi service layer.
  10. Đảm bảo current user được dựng từ trusted headers/internal JWT qua CurrentUser; không nhận userId từ request body cho API của chính user hiện tại.
  11. Implement hoặc hoàn thiện GET /api/v1/users/me: trả id, email, displayName, avatarUrl, bio, role/roles, status; không trả password, internal token hoặc field audit nhạy cảm.
  12. Implement hoặc hoàn thiện PUT /api/v1/users/me/profile: validate displayName không rỗng, avatarUrl hợp lệ nếu có, bio giới hạn độ dài; update đúng user đang login.
  13. Nếu cần DB change, thêm Liquibase migration nhỏ vào user-management-service, ví dụ thêm bio hoặc avatar_url; đặt default/nullability rõ ràng.
  14. Xử lý lỗi: thiếu token trả 401 ở gateway, token sai audience/issuer trả 401, user DISABLED trả 403 hoặc response status rõ theo rule service.
  15. Evidence bắt buộc: login bằng student gọi /api/v1/users/me, update profile rồi read-back; gọi không token để thấy 401; dùng token admin để thấy role khác student.

Kết quả cần đạt

  • Admin web login bằng admin@courseflow.local/password.
  • Learner web/mobile app login bằng student@courseflow.local/password.
  • /api/v1/users/me trả đúng id, email, role, status.
  • Request không token trả 401.
  • Client tự gửi X-User-Id giả không thay đổi identity thật.

Evidence

GET http://localhost:28080/api/v1/users/me
Authorization: Bearer <keycloak_access_token>

Expected:
- HTTP 200
- id/email/role đúng user login
- response không chứa password/token/internal secret

Negative:
- Không có Authorization -> 401
- Gửi thêm X-User-Id: 999 -> service vẫn nhận user thật từ gateway

OIDC/JWKS evidence:
- Chụp issuer và jwks_uri từ discovery document.
- Chụp JWKS có key dạng RSA/RS256 hoặc public signing key.
- Chỉ ra code gateway dùng jwk-set-uri để verify token.
- Chỉ ra internal JWT sau gateway mới dùng HS256 training secret.
