Ngày 11: Notification và outbox

Dùng event/outbox để tạo notification sau các nghiệp vụ chính.

Notification Outbox Idempotency

Sơ đồ impl

flowchart LR Business["Enrollment/Grade transaction"] --> DomainDB[("domain DB update")] Business --> Outbox[("outbox event
same transaction")] Outbox --> Relay["outbox-relay"] Relay --> Kafka["Kafka topic"] Kafka --> Consumer["notification consumer"] Consumer --> Idem["eventId idempotency"] Idem --> NotificationDB[("notification DB")] NotificationDB --> API["list / mark read"] LearnerWeb["Learner web"] --> API MobileApp["Mobile app"] --> API

Mục tiêu

  • Business transaction ghi outbox event.
  • Consumer/relay tạo notification từ event.
  • User list và mark read notification.

Cách impl

  1. Đọc notification-service và outbox-relay: outbox table, relay polling, Kafka topic, notification inbox/read state.
  2. Chốt event nào tạo notification trong training: enrollment.created, submission.graded, quiz.completed hoặc course.published.
  3. Trong producer service, ghi outbox cùng transaction với business change; không publish Kafka trực tiếp trước khi DB commit.
  4. Outbox record cần có aggregateId, aggregateType, eventType, payload JSON, createdAt, publishedAt/null, retry/dead-letter nếu đã có schema.
  5. outbox-relay poll unpublished theo batch nhỏ, publish Kafka, mark published; nếu publish fail thì retry và không mất record.
  6. Consumer notification-service parse event, build notification target user, title, body, type, link, eventId.
  7. Idempotency consumer: eventId unique hoặc processed marker; Kafka redelivery không tạo duplicate notification.
  8. List my notifications: current user only, paging, unread first hoặc createdAt desc.
  9. Mark read: chỉ owner được mark; idempotent nếu đã read.
  10. Unread count endpoint nếu web cần badge; count theo userId + readAt null.
  11. Error handling: event payload thiếu field quan trọng thì log + dead-letter/skip có reason, không crash loop vô hạn.
  12. Evidence bắt buộc: grade submission sinh notification, list thấy unread, mark read giảm count, replay cùng event không duplicate.

Kết quả cần đạt

  • Enroll course tạo notification cho learner.
  • Grade update tạo notification cho learner.
  • List notification có unread/read state.
  • Xử lý lại cùng event không tạo notification trùng.

Evidence

Trigger: enroll course hoặc update grade
Expected:
- outbox có event
- notification-service có notification cho learner

GET /api/v1/notifications
Authorization: Bearer <student_token>
Expected: notification mới

POST /api/v1/notifications/{id}/read
Expected: readAt được set

Negative:
- replay eventId cũ -> không duplicate
