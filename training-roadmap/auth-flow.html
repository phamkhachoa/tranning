Luồng AuthN/AuthZ đầy đủ

CourseFlow Mini dạy auth theo stack thật: web/app login bằng Keycloak/OIDC, gateway verify access token, rồi downstream service chỉ tin internal JWT và trusted headers do gateway set. Để giảm độ khó, bản training bỏ STS riêng và cho gateway tự sinh internal JWT local.

Từ JWT secret sang OAuth2/OIDC JWKS

Trước khóa này, các bạn có thể đã quen với bài đơn giản: backend tự ký JWT bằng một secret key, rồi chính backend verify lại bằng cùng secret đó. Cách đó phù hợp để học JWT format, nhưng không phải mô hình chuẩn cho web/app đăng nhập qua một Identity Provider như Keycloak.

Tiêu chí JWT HS256 bằng secret key OAuth2/OIDC access token bằng JWKS
Ai ký token? Chính backend hoặc auth-service nội bộ. Keycloak ký token bằng private key của realm.
Ai verify? Service nào có shared secret đều verify được. API Gateway tải public key từ JWKS endpoint của Keycloak để verify chữ ký.
Key dùng để verify Cùng một secret dùng cho cả ký và verify. Public key theo kid trong JWT header; private key không rời Keycloak.
Rủi ro chính Lộ secret là có thể tự ký token giả. Service chỉ có public key nên không thể tự ký token Keycloak giả.
Phù hợp để học gì? JWT cơ bản, signature, expiry. Auth thật: issuer, audience, key rotation, client login, SSO.
Quy tắc bắt buộc: external token từ Keycloak phải verify bằng issuer + audience + JWKS/public key. Không được quay lại kiểu tự decode payload rồi tin claim, cũng không dùng shared secret để verify token Keycloak.

API Gateway verify Keycloak token như thế nào?

sequenceDiagram autonumber participant C as Web/App participant KC as Keycloak Realm participant GW as API Gateway participant JWKS as Keycloak JWKS participant DS as Downstream Service C->>KC: Authorization Code + PKCE KC-->>C: access_token RS256 C->>GW: Authorization: Bearer access_token GW->>GW: read JWT header kid/alg GW->>JWKS: GET /protocol/openid-connect/certs JWKS-->>GW: public keys GW->>GW: select key by kid and verify signature GW->>GW: validate iss, aud, exp, sub GW->>GW: resolve user/roles and mint internal JWT GW->>DS: X-Internal-Authorization + trusted X-User-*
  1. Keycloak phát hành access token. Token thường dùng asymmetric signing như RS256: private key ký, public key verify.
  2. JWT header có kid. Gateway dùng kid để chọn đúng public key trong JWKS.
  3. Gateway dùng NimbusReactiveJwtDecoder.withJwkSetUri(...) để tải/cache JWKS và verify chữ ký.
  4. Sau chữ ký, gateway vẫn phải validate iss, aud, exp, sub. Verify signature đúng nhưng sai audience vẫn phải reject.
  5. Gateway không lấy role client gửi qua header. Nó xóa mọi X-User-* từ client rồi tự set lại trusted headers.
  6. Trong training, TOKEN_CONVERTER_MODE=local: sau khi verify token Keycloak, gateway tự sinh internal JWT HS256 cho downstream service.
External token:
Keycloak private key --sign--> access_token
API Gateway JWKS public key --verify--> accepted/rejected

Internal token trong mini stack:
API Gateway local HS256 secret --sign--> X-Internal-Authorization
Downstream service same internal secret --verify--> CurrentUser

Lệnh kiểm chứng bắt buộc

# 1. Xem public keys mà gateway dùng để verify Keycloak token
curl http://localhost:18080/realms/courseflow/protocol/openid-connect/certs

# 2. Xem issuer và jwks_uri từ OIDC discovery document
curl http://localhost:18080/realms/courseflow/.well-known/openid-configuration

# 3. Xem runtime env của gateway trong training stack
docker compose -f backend/infra/docker/docker-compose.training.yml exec -T api-gateway sh -c \
  'env | sort | grep -E "COURSEFLOW_TRAINING_AUTH_BYPASS|TOKEN_CONVERTER_MODE|KEYCLOAK_ISSUER_URI|KEYCLOAK_JWK_SET_URI|KEYCLOAK_AUDIENCE|COURSEFLOW_INTERNAL_JWT_ALGORITHM"'

Kết quả mong đợi trong flow thật:

COURSEFLOW_TRAINING_AUTH_BYPASS=false
TOKEN_CONVERTER_MODE=local
KEYCLOAK_AUDIENCE=courseflow-api
KEYCLOAK_JWK_SET_URI=http://keycloak:8080/realms/courseflow/protocol/openid-connect/certs
COURSEFLOW_INTERNAL_JWT_ALGORITHM=HS256
Lưu ý địa chỉ: trình duyệt thấy Keycloak qua localhost:18080, nhưng container gateway gọi Keycloak bằng DNS nội bộ Docker keycloak:8080. Hai URL này cùng trỏ tới realm courseflow trong môi trường khác nhau.

Keycloak training flow

sequenceDiagram autonumber actor U as User participant W as Web/App Client participant KC as Keycloak participant GW as API Gateway participant UM as user-management participant AC as access-control participant DS as Domain Service U->>W: Click login on web/app W->>KC: Authorization Code + PKCE KC-->>W: authorization code W->>KC: exchange code KC-->>W: access token W->>GW: API request + Bearer token GW->>KC: fetch JWKS if needed GW->>GW: verify issuer + audience + signature GW->>GW: map demo email to userId/role GW->>GW: mint local internal JWT GW->>DS: X-User-* + X-Internal-Authorization DS->>DS: verify internal JWT DS->>AC: permission/resource check AC-->>DS: allow/deny + audit DS-->>GW: response GW-->>W: response 
1. User mở Admin web, Learner web hoặc Mobile app.
2. User bấm đăng nhập.
3. Web/app dùng Authorization Code + PKCE redirect sang Keycloak.
4. Keycloak xác thực password và trả authorization code.
5. Web/app đổi code lấy access token.
6. Web/app gọi API Gateway với Authorization: Bearer <external_access_token>.
7. API Gateway verify JWT bằng issuer, audience và JWKS của Keycloak.
8. Gateway map demo user từ Keycloak claim sang user id/role trong database training.
9. Gateway tự sinh internal JWT ngắn hạn bằng local signing key của mini stack.
10. Gateway xóa mọi header X-User-* từ client rồi set lại X-User-Id, X-User-Roles, X-Internal-Authorization.
11. Downstream service chỉ tin request có internal JWT hợp lệ.
12. Controller nhận CurrentUser từ common-library.
13. Service kiểm tra quyền: role/permission qua access-control-service và resource ownership/enrollment/staff access trong domain.
14. Access-control ghi audit decision: subject, permission, scope, allowed/denied, reason.
Giản lược có chủ đích: production có thể dùng STS/token-converter riêng. Training dùng TOKEN_CONVERTER_MODE=local để học đúng boundary mà không phải vận hành thêm một service auth.

Stack local cần chạy

cd backend/infra/docker
docker compose -f docker-compose.training.yml up -d --build

Keycloak chạy tại http://localhost:18080. Admin console dùng admin/admin. Web admin dùng client courseflow-admin-web; learner web dùng client courseflow-learner-web; mobile app dùng client courseflow-mobile. Cả ba client cùng realm courseflow và đều gọi cùng API Gateway.

Fallback bypass flow

Chế độ này chỉ dùng khi cần tiếp tục lớp học dù máy học viên chưa dựng được Keycloak. Bật bằng COURSEFLOW_TRAINING_AUTH_BYPASS=true. 

1. Người học tạo session demo trong web localStorage hoặc app dev storage.
2. Web/app gửi Authorization: Bearer training và X-Training-User-*.
3. Gateway bỏ qua verify Keycloak vì COURSEFLOW_TRAINING_AUTH_BYPASS=true.
4. Gateway vẫn xóa X-User-* giả mạo từ client.
5. Gateway lấy X-Training-User-* để sinh internal identity trusted.
6. Downstream service vẫn nhận CurrentUser và vẫn chạy authorization như bình thường.
7. Forbidden/allowed evidence vẫn phải chứng minh qua API.
Điểm cần kiểm tra: training bypass chỉ được tồn tại ở edge/gateway. Domain service không được tự đọc X-Training-User-* và không được bỏ qua authorization.

  • Cùng browser profile có thể mở admin web, learner web và app cùng lúc, nhưng chúng dùng chung SSO user trong realm courseflow.
  • Muốn test hai user khác nhau đồng thời, dùng browser profile khác nhau hoặc normal window + incognito.
  • Nếu đang login admin@courseflow.local rồi mở learner /register, Keycloak có thể báo different_user_authenticated. Đây là rule SSO đúng, không phải lỗi pod/frontend.
  • Admin tạo learner bằng POST /api/admin/v1/users; learner self-registration chỉ dành cho visitor chưa có SSO session của user khác.

Tạo tài khoản learner

  1. Luồng backoffice: admin web gọi POST /api/admin/v1/users; gateway rewrite sang user-management /backoffice/admin-users.
  2. User-management gọi Keycloak Admin API để tạo account với mật khẩu tạm thời hoặc setup-email nếu đã có SMTP.
  3. User-management gọi access-control để link issuer + subject, gán role mặc định STUDENT, rồi tạo profile mặc định.
  4. Luồng learner web: /register mở Keycloak registration bằng Authorization Code + PKCE.
  5. User self-register login xong, gateway local-mode resolve/provision identity qua access-control và gán STUDENT nếu đây là lần đầu user xuất hiện.
  6. Mọi client web/app sau đó gọi API bằng access token Keycloak; service chỉ tin internal JWT/trusted headers từ gateway.
Bài học chính: flow này thuộc Ngày 03 - User admin và profile. Ngày 01 chỉ dạy login/current user với tài khoản seed; Ngày 02 dạy RBAC/permission; Ngày 03 ghép user lifecycle đầy đủ.
Cách test đúng: admin provision learner trên admin web trong browser thường. Self-registration learner nên test bằng incognito/browser profile khác nếu browser thường đang login admin.

Authorization evidence bắt buộc

  1. Unauthenticated/protected request bị chặn khi thiếu Keycloak session hoặc thiếu bearer token.
  2. Student gọi admin endpoint phải bị 403 Forbidden.
  3. Instructor thao tác đúng course của mình được phép.
  4. User không được tự gửi X-User-Id để giả danh người khác.
  5. Endpoint nhạy cảm có audit/log quyết định quyền: subject, action, resource, reason.
