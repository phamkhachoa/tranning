Không cho full schema, cho rule để tự thiết kế

Người học tự đề xuất table, column, index và constraint. Review nhanh bằng câu hỏi: rule nào được bảo vệ ở service layer, rule nào được bảo vệ ở database?

Checklist trước khi code migration

  1. Liệt kê aggregate chính và owner service.
  2. Liệt kê business invariant cần DB constraint bảo vệ.
  3. Chọn khóa chính, unique key, foreign key hoặc lý do không dùng FK qua service boundary.
  4. Chọn index cho query list/search phổ biến.
  5. Định nghĩa status enum và transition hợp lệ.
  6. Có audit columns: created_at, updated_at, created_by nếu cần.

Ví dụ guardrail theo feature

Feature Rule cần giữ Người học tự đề xuất
Course Slug unique; public chỉ thấy PUBLISHED; draft thuộc instructor. Bảng course, course_status, unique slug, index status/category.
Enrollment Một learner không có hai ACTIVE enrollment trong cùng course. Unique partial index hoặc constraint + service idempotency.
Assignment Submission thuộc learner đã enroll; score nằm trong 0..maxScore. Bảng assignment, submission, status, score constraint.
Chat Message thuộc course room; history query phải nhanh theo courseId + createdAt. Mongo collection chat_rooms/chat_messages, compound index courseId + createdAt.
Quiz Attempt chỉ submit một lần; không leak đáp án đúng. Bảng quiz, question, option, attempt, attempt_answer.
Gradebook Một grade item không được nhân đôi khi recalculate. Unique key learner/course/source/source_id.
Notification Event xử lý lại không tạo notification trùng. Unique event_id hoặc dedup key.
Search Không query SQL LIKE cho public search/recommend; dùng Debezium CDC để project course DB sang read model riêng. Postgres logical replication, Debezium connector, Kafka topic, Elasticsearch document, analyzer, keyword facets.
Outbox vs CDC Outbox dùng cho business event có ý nghĩa nghiệp vụ; CDC dùng để đồng bộ read model theo thay đổi DB. Không polling bảng. Không dùng CDC event thô thay cho audit/business command nếu cần domain intent.

Mẫu nộp DB proposal

Feature: Enrollment

Tables:
- enrollments(id, learner_id, course_id, status, enrolled_at, completed_at, created_at, updated_at)

Constraints:
- status in ACTIVE, COMPLETED, CANCELLED
- unique active enrollment per learner/course

Indexes:
- learner_id + status for my learning dashboard
- course_id + status for instructor roster

Open question:
- Có cần FK tới course không? Nếu course nằm service khác, chỉ lưu course_id và validate qua API.
